**Заметки:** Вот различие, которое я хочу обозначить. ИИ делает сложность терпимой. Он даёт вам лучший интерфейс для работы со сложными вещами. Но он не снимает необходимость с ними работать. Настоящая автоматизация — та, что сделала Heroku мощным в своё время, та, что делает Vercel тем, чем он является сегодня, — не делает сложность терпимой. Она снимает необходимость вообще о ней думать. Вы не спрашиваете ИИ, как настроить деплой на Vercel. Вы об этом не думаете. Оно просто работает. Это принципиально разные вещи. ## СЛАЙД 13 — Вопрос, который возник
**На слайде:**
```
Почему у фронтендерского мира есть это,
а у бэкендерского — нет?
```

**Заметки:** Почему в фронтенд-экосистеме есть платформы, которые просто берут это на себя, — а в бэкенд-экосистеме нет? Хочу быть точным — я понимаю, что некоторые из вас уже придумывают контраргументы. Да, решения существуют. Laravel Cloud запустился в феврале 2025 — zero-config, managed БД, автомасштабирование, DevOps не нужен. Deno Deploy — zero-config для JS и TypeScript. В Rails встроен Kamal с версии 8. Всё это реально, и некоторые из них действительно впечатляют. Но обратите внимание на паттерн. Каждая экосистема построила своё решение, для себя, изолированно. Laravel Cloud — только для Laravel. Deno Deploy — только для Deno. Kamal требует своего сервера и знания Docker. Это не универсальные стандарты — это каждое сообщество в одиночку решает одну и ту же проблему. У Go-разработчиков этого нет. У Python нет. У Java нет. У Rust нет. И даже там, где есть, этого достаточно разрозненно, чтобы разработчику всё равно пришлось самому найти, оценить и понять, подходит ли. Это бремя принятия решений — то самое, что мы пытаемся убрать.

## СЛАЙД 14 — Карта экосистем
**На слайде:**
```
JavaScript / TS  →  Vercel, Deno Deploy      ✓
PHP / Laravel    →  Laravel Cloud             ✓
Ruby / Rails     →  Kamal (свой сервер)       ⚠

Go               →                            ✗
Python           →                            ✗
Java             →                            ✗
Rust             →                            ✗
```

**Заметки:** Посмотрим честно. Какие экосистемы решили эту задачу — а какие нет. JavaScript и TypeScript — Vercel и Deno Deploy. Zero-config, push to deploy, никаких мыслей об инфраструктуре. PHP и Laravel — Laravel Cloud с февраля 2025 года: полностью управляемый, zero-config, сделан самой командой Laravel. Ruby — Kamal в составе Rails 8: значительно упрощает процесс, но вы всё равно приносите свой сервер и всё равно должны знать Docker. Ну а Go, Python, Java, Rust — ничего. Нет официальной платформы, нет стандарта сообщества, нет zero-config пути от людей, которые лучше всех знают стек. Заметьте паттерн. Экосистемы, которые решили эту проблему, объединяет одно: сильное централизованное руководство, которое решило взять историю с деплоем в свои руки. Команда Go в Google никогда этого не делала. Сообщество Python никогда этого не делало. Поэтому разработчики остаются наедине с универсальными облачными платформами, которые не понимают их стек. Этот разрыв реален. И именно там живёт боль.

## СЛАЙД 16 — Почему фронтенд ощущается иначе
**На слайде:**
```
Vercel не упростил деплой.

Он сделал его невидимым.
```

**Заметки:** Когда я думаю о том, почему фронтендерская доставка ощущается лёгкой, я снова и снова возвращаюсь к этому различию. Vercel не просто упростил деплой — он сделал его чем-то, о чём вы не думаете. А это разные вещи. Упростить что-то — значит, вы всё равно это делаете, просто с меньшим трением. Сделать невидимым — значит, это больше не занимает место в вашей голове вообще. Джуниор-разработчик, деплоящий свой первый Next.js-проект, не думает о конфигурации CDN, граничном кешировании, оптимизации сборки или настройке окружения. Всё это происходит — это реально и важно, — но невидимо. Работа разработчика — писать код и пушить его. Всё остальное просто работает. Как им это удалось? Три принципа. И я думаю, эти принципы применимы к бэкенд-доставке тоже.

---

## СЛАЙД 17 — Принцип 1: Определяй
**На слайде:**
```
Автоопределение.

Платформа сама разбирается, что у тебя есть.
Ты ей не говоришь.
```

**Заметки:** Первый принцип — автоопределение. Вы пушите Next.js-проект — платформа знает, что это Next.js. Пушите Remix, Astro, SvelteKit — она их распознаёт. Вы не пишете конфиг-файл, который говорит «это Next.js-проект, пожалуйста, используй пайплайн для Next.js». Платформа читает ваш проект и сама разбирается. Это звучит мелко. На самом деле нет. Потому что это значит, что первое действие разработчика при желании что-то задеплоить — это написание кода, а не написание конфигурации, описывающей код, который он собирается написать. Отсутствие этого первого шага с конфигурацией меняет весь опыт онбординга. Это разница между «дай покажу, как это работает» и «прежде чем начать, тебе нужно заполнить эту форму».

---

## СЛАЙД 18 — Принцип 2: Решай
**На слайде:**
```
Zero-config по умолчанию.

Разумные решения приняты один раз,
а не каждым разработчиком каждый раз.
```

**Заметки:** Второй принцип — zero-config по умолчанию. Как только платформа знает, что у вас есть, она принимает решения за вас. Команда сборки, выходная директория, настройка окружения, стратегия кеширования — всё это. Это решения, которые кто-то принял один раз, тщательно, исходя из того, что работает для большинства проектов. И каждый разработчик, который приходит после, получает выгоду от этих решений, не зная об их существовании. Это тонко, но важно. Это значит, что коллективное знание «как правильно задеплоить Next.js-приложение» живёт в платформе, а не на странице в вики, не в племенном знании, которое есть только у сеньора, не в сорокашаговом онбординг-документе. Оно закодировано. Оно автоматично. И когда вам нужно переопределить — вы можете. Но вам не надо понимать его, чтобы начать.

## СЛАЙД 19 — Принцип 3: Откладывай
**На слайде:**
```
Прогрессивная сложность.

Простые вещи — просто.
Продвинутые — возможны, но не обязательны.
```

**Заметки:** Третий принцип — прогрессивная сложность. Если вам нужна кастомная конфигурация — она есть. Edge-функции, кастомные заголовки, продвинутые правила кеширования, монорепо — всё доступно. Но вам не нужно ничего из этого понимать, прежде чем задеплоить первый проект. Сложность откладывается до тех пор, пока она вам реально не понадобится. Именно над этим я думаю больше всего — потому что именно здесь бэкендерский мир чаще всего ошибается. На бэкенде сложность стоит в начале. Нельзя задеплоить, не разобравшись сначала в IAM, реестрах, пайплайнах. Продвинутые вещи не опциональны — они обязательны как предварительное условие. Прогрессивная сложность переворачивает это. Начни просто. Задеплой что-нибудь. Разбирайся в глубоких слоях, когда появится причина — когда у вас есть работающий сервис и реальная проблема для решения.

И я хочу быть честным — ни один из этих инструментов не плох. Cloud Run — действительно отличная инфраструктура, когда вы в ней разобрались. Но «когда вы в ней разобрались» делает много работы в этом предложении. Разобраться в ней — это целая работа до того, как начнётся настоящая работа.

## СЛАЙД 23 — Сравнение
**На слайде:**
```
Go    →  единый бинарник  →  Lambda ✓   Cloud Run ✓
Node  →  SSR / рантайм    →  Lambda ⚠   Cloud Run ✓
Ruby  →  нужен рантайм    →  Lambda ✗   Cloud Run ✓
PHP   →  нужен рантайм    →  Lambda ✗   Cloud Run ✓
```

**Заметки:** Go компилируется в единый бинарник. Никаких рантайм-зависимостей. Быстрый холодный старт. Почти идеально для Lambda — запакуй и готово. Cloud Run тоже отлично работает. Реальные варианты, оба разумны. Node с серверным рендерингом — другая история. Можно затащить в Lambda, но нужен слой адаптера — это неестественно, добавляет сложности. Cloud Run справляется чисто, потому что просто запускает контейнер и не заботится о рантайме. Ruby и PHP нужны кастомный рантайм или Docker-упаковка для Lambda. Технически возможно, но не лёгкий путь — это отдельный проект. Cloud Run работает, потому что снова — просто контейнер. Суть не в том, что одна платформа лучше. Суть в том, что это решение — какая платформа подходит вашему стеку — можно автоматизировать. Инструмент, который определяет ваш стек, может применить правильный дефолт, предложить правильную цель. Это знание должно жить в инструментах, а не в голове разработчика в первый день.

## СЛАЙД 25 — Правильный порядок
**На слайде:**
```
Порядок, в котором ты встречаешь вещи,
определяет, как ты их понимаешь.
```

**Заметки:** Хочу предложить другой порядок обучения. Не потому что инфраструктура не важна — она абсолютно важна. Разработчик, который понимает свою инфраструктуру, — лучший разработчик. Но порядок, в котором вы встречаете вещи, определяет, как вы их понимаете. Разработчик, который тратит первую неделю на написание бизнес-логики — на создание чего-то, что реально что-то делает, решает реальную задачу, — имеет совершенно иное отношение к инфраструктуре, когда в конце концов до неё доходит. Он понимает, зачем она нужна. У него есть работающий сервис и реальная проблема для решения. Сложность имеет контекст. В отличие от разработчика, который тратит первую неделю на борьбу с IAM-правами, не написав ни одной осмысленной строчки кода. Этот разработчик много узнал об облачных платформах — и почти ничего о проектировании ПО.

**На слайде:**

Это знание теперь имеет контекст. День девяностый — вы готовы владеть инфраструктурой, принимать осознанные решения, может быть, даже оспаривать дефолты. Вы заработали сложность, потому что понимаете, для чего она. Прогрессивная сложность — это не о том, чтобы скрывать вещи навсегда. Это о правильном времени. Фронтендерские инструменты это поняли. Бэкенд-деплой в основном — ещё нет.

## СЛАЙД 27 — Стандарт
**На слайде:**
```
Этот инструмент убирает проблему?

Или он просто делает проблему
легче терпеть?
```

**Заметки:** Я возвращаюсь к этому вопросу каждый раз, когда смотрю на инструменты для разработчиков. Не «хорошо ли этот инструмент спроектирован» — как правило, да. Не «лучше ли он альтернатив» — возможно. Но убирает ли этот инструмент проблему, или просто даёт мне лучший интерфейс для работы с ней? Когда make автоматизировал компиляцию, разработчики перестали думать о компиляции. Когда npm автоматизировал управление зависимостями, разработчики перестали отслеживать версии библиотек. Проблема исчезла. Вот ориентир. И я думаю, мы его понизили. Мы начали принимать «легче терпеть» за «достаточно хорошо». ИИ — самый свежий пример: он делает сложность плохого DX более терпимой. Но терпимо — это не то же самое, что исчезло. Я пришёл из фронтенда, где ориентир был выше. И как только я это заметил, не мог перестать видеть.
